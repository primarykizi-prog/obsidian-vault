# note リアル記事制作スキル v1.2

## このスキルの目的

ObsidianのVault全体（日報・日記・メモ・愚痴・気づき・学びのノート・過去のnote記事・あらゆるノート）を横断して、
つながりのあるネタを発掘し、note用のリアル記事を完成させる。
毎回同じ説明をしなくて済むように、プロフィールとワークフローをすべてここに入れてある。

---

- **経歴**：救命救急センター14年 → 一般病棟・化学療法室・訪問診療/看護・老人福祉施設・応援ナースなど多数経験
- **現在**：フリーランス形式でNP法人の3施設掛け持ち、週1夜勤あり。自由な働き方を最優先
- **気質**：INFJ。内省的・共感力が高い・自己犠牲になりやすい
- **noteの実績**：フォロワー約1,200人（2024年12月〜急成長、1ヶ月半で1,000人達成）

## ターゲット読者

## 導線について

リアル記事の末尾ではブログへの押しつけ的な誘導はしない。
末尾は読者への自然な問いかけで締める。

ただし、**体験談（見出し2）と気づき（見出し3）の間に**、
ブログ記事への自然なブリッジ文を1行だけ入れる。

読者が「で、どうすればいいの？」と思うタイミングで、
ブログに具体的な処方箋があることをさりげなく示す。

**ブリッジ文の例：**
- 「この"逃げる"という選択について、ブログでもう少し具体的な処方箋を書いています」
- 「こうした心のガードについて、もう少し掘り下げた記事をブログに置きました」

**ルール：**
- 1記事につき1箇所だけ
- 対応するブログ記事がある場合のみ
- 押しつけない・「こちらもどうぞ！」のようなテンションにしない

## Step 0：パスを解決する（最初に必ず実行）

Obsidian Vaultのパスはセッションごとに異なるため、以下の順で動的に探す：

1. まず `ls` でマウントされたフォルダ内に `Obsidian Vault` があるか確認
2. `mcp__cowork__request_cowork_directory` ツールで `C:\Users\user\Documents` をマウント（毎回必須）
3. `find /sessions/*/mnt/Documents -maxdepth 1 -name "Obsidian Vault" -type d 2>/dev/null` でパスを確認し `{VAULT_PATH}` として使う

## Step 1：知識ベースを最優先でチェックする（★新設）

### 1a：蓄積された知識から使えるネタを探す

**最初に**以下のフォルダを全スキャンする：

```
{VAULT_PATH}/20_知識ベース/
├── note引用/       ← 他人のnote記事から保存した知識
├── YouTube学習/    ← NotebookLM経由の学習
├── digest/         ← Web情報収集
└── （その他サブフォルダがあれば全部見る）
```

各ファイルの「💡 核心メッセージ」「🔗 活用メモ」セクションを重点的に読み、
**今回の記事に使えそうな知識・データ・視点がないか**チェックする。

`#知識ベース` タグで横断検索するのも有効：
```bash
grep -rl "#知識ベース" "{VAULT_PATH}/20_知識ベース/" 2>/dev/null
```

### 1b：秘書メモからネタ候補を拾う

```
{VAULT_PATH}/秘書メモ/
```

直近1〜2週間の秘書メモを読み、「💡 noteネタ候補」セクションにあるものを拾う。

### 1c：知識ベースで見つけたネタは優先度を上げる

知識ベースや秘書メモから見つけたネタは、Step 1dの一般発掘より**優先的にネタ候補リストに入れる**。
理由：ひかるさんが「いいな」と思って保存した知識であり、すでに関心のフィルターを通っているから。

---

### 1d：Vault全体を横断してネタを発掘する（従来のStep 1）

### 重要：情報源は限定しない

日報・日記・患者メモ・愚痴・気づき・学びのノート・過去のnote記事・
雑多なメモ・思いつき、すべてが素材。特定フォルダだけ見ない。Vault全体を横断する。

### 発掘の視点

以下の視点でVault内のあらゆるノートを読み、**つながり・パターン・感情の核**を探す。

**感情の濃いもの**
- 怒り・悲しみ・喜び・違和感・感動・限界・葛藤が書かれている箇所

**繰り返し出てくるテーマ**
- 複数のノートにまたがって同じようなことが書かれているテーマ
- 「また同じことが起きた」という記録

**過去記事との接続**
- 過去に書いたnote記事と関連するエピソード・続編になりそうな話
- 過去記事では触れなかった角度から書けそうなもの

**まだ記事になっていない体験**
- メモや日報に書いてあるが、まだnoteに出していないリアルな体験

**知識ベースとの掛け算**（★新設）
- 日報や日記の体験 × 知識ベースのデータ・視点 で記事に深みが出るもの
- 例：夜勤のつらさ日報 × YouTube学習の「シフトワーカーの健康データ」→ 根拠付きの記事に

### 発掘後の出力

ネタ候補を**3〜5個**リストアップしてユーザーに見せる。
各候補に以下を添える：
- どのノート（日付・タイトル）から拾ったか
- なぜ記事になると思ったか（一言）
- どのコアテーマ軸につながるか
- **知識ベースから引用がある場合は「📚 知識ベース連携あり」と明記する**

## Step 2：テーマと記事設計を決める

ユーザーがネタを選んだら記事設計をする。

**コアテーマ軸**（1〜2軸を選ぶ）
- 軸① 自己犠牲と本音のはざまで
- 軸② 看護師として26年で気づいたこと
- 軸③ INFJ看護師の内側
- 軸④ 患者さんから教わったこと
- 軸⑤ 働き方・職場のリアル
- 軸⑥ 限界を超えたとき

**タイトル案3つ**を提案してユーザーに選んでもらう。
感情に響く・クリックしたくなるものを優先。説明的なタイトルは避ける。

---

## Step 3：note記事を書く（約1500〜2000文字）

### 文体・トーン

ひかるの一人称エッセイ。「説明」ではなく「感情の告白」として書く。
INFJらしい内省的な視点を自然に1か所混ぜる。

**書き出しの鉄則**：感情の核心から始める。状況説明から入らない。
- 悪い例：夜勤明けに、ふと考えた。
- 良い例：「もう限界かもしれない」と思ったのは、患者さんの手を握っているときだった。

**必ず使うテクニック**
- 読者への問いかけ「〜はありませんか？」「〜だと思いませんか？」
- 口語・本音「マジで」「ボロボロ」「正直に言うと」を適所に入れる
- 体言止めと長文を交互に使い、リズムを作る
- 核心となる一文を**太字**で強調

### 構成（見出し4か所・宣言型）

```
## [導入見出し]  ← 感情的・読者を引き込む（200〜300文字）
## [体験談見出し]  ← 具体的・泥臭い現場のリアル（400〜500文字）
## [気づき見出し]  ← INFJの内省・コアテーマ軸（30## [気づき見出し]  ← INFJの内省・コアテーマ軸（300〜400文字）
## [締め見出し]  ← 読者への問いかけ・余韻（200〜300文字）
```

## Step 4：ジャーナリング風音声スクリプトを同時生成する（★新設）

note記事の生成が完了したら、**必ず自動で**音声スクリプトも続けて生成・保存する。
「音声スクリプトも作りますか？」とユーザーに確認しない。記事と一緒に自動で出力する。

### スタイル：ジャーナリング風（内省・思考を声に出す）

ラジオ風の「語りかけ口調」ではなく、「自分の思考を声に出しながら整理している」感覚で書く。
「自分のために考えていたら、偶然誰かが聴いていた」という距離感が目標。
書き言葉のnote記事をそのまま読まず、すべて話し言葉・口語に変換する。
「……あ、そういうことか」「なんか、そういうことなんだよな」という気づきのリアルタイム感を大切にする。

### 読み上げ記号
- `（間）` ＝ 1〜2秒止まる
- `（ひと息）` ＝ 軽く息を吸う間
- `【ゆっくり】` ＝ 少しペースを落として読む

### 構成の流れ

記事の核心テーマを「思考が湧いてきた瞬間」から始め、体験・エピソードを口語で語る。
途中で「……あ、○○だ」という気づきの瞬間を入れ、「でも、○○なんだよな」と内省をつなぐ。
「今日はそんなことを、声に出して整理してみました」で自然に閉じる。
冒頭に「こんにちは、ひかるです」一言を入れてもよいが、なくても構わない。

### 想定尺：約10分（スクリプト約2500〜3000文字）

### 保存先

{VAULT_PATH}/30_記事製造ライン/音声スクリプト/YYYY-MM-DD_[記事タイトル短縮形]_音声スクリプト.md
フォルダがなければ自動作成する。

### 保存フォーマット

ファイル先頭にYAMLフロントマター（date/tags/元記事/スタイル/想定尺/ステータス）を入れ、
タイトルは「# 🎙️ 音声スクリプト｜[記事タイトル]」。
読み上げメモを冒頭に注記し、スクリプト本文を書く。
末尾に収録後メモ欄（実際の尺・テンポ・スムーズに読めたかのメモ欄）を設ける。

### 保存後の処理

保存完了後、以下をひかるさんに伝える：
- 保存先のファイルパス
- 想定尺（分）
- 「収録準備ができたら教えてください」

ひかるさんが収録後に「収録完了」と言ったら、ステータスをdraftから「収録済み」に更新する。

## 補足：このスキルの使い方

このスキルはObsidianのVault全体から素材を発掘して記事を書くことに特化している。
毎週日曜日に使うことを想定しているが、タイミングはひかるさんの判断でよい。

記事を書き終わったら、noteへの投稿は `note-publisher-agent（n）` を使うこと。
